# WPS `DISPIMG` 工作流

## 当前结论

项目的图片方案正式按 `WPS` 优先处理。

目标不是：

- `Microsoft Excel` 完整兼容
- 标准 xlsx 图片语义统一

目标是：

- 列表工作簿主要给 `WPS` 使用
- 图片最终在 `WPS` 里表现成嵌入单元格图片
- 后续尽量落到 `DISPIMG`

## 当前程序已经做好的事

保存列表 Excel 时，程序已经会：

1. 生成 `商品列表.xlsx`
2. 在主表里保留 `图片路径`
3. 把图片复制到 Excel 同级 `img/` 目录
4. 生成：
   - `商品列表`
   - `图片关系`
   - `变体关系`
   - `扩展字段`
   - `程序信息`

也就是说，WPS 宏需要的输入已经齐了。

## 为什么还不能直接由程序写出 `DISPIMG`

因为难点不是公式本身，而是：

- `DISPIMG("ID_xxx", 1)` 里的 `ID_xxx` 怎么生成
- 图片资源如何注册到 WPS 可识别的结构
- 这套结构目前没有稳定、完整、公开的可编程文档

所以当前不建议：

- 在 Rust / Node 里直接伪造 `DISPIMG` 内部资源结构

## 推荐工作流

建议按 3 步走：

### 第 1 步：程序导出列表工作簿

结果：

- 主表有 `图片路径`
- Excel 同级有 `img/`

### 第 2 步：在 WPS 中执行宏

宏负责：

- 读取主表 `图片路径`
- 找到每行对应图片
- 在图片列插入图片并与单元格对齐

### 第 3 步：由 WPS 完成“嵌入单元格图片”

当前最稳的方式仍然是：

- 让 WPS 自己完成最后一步转换

如果后面确认存在稳定的宏接口，可以再把这一步也写进宏。

## 为什么先接受这个阶段性方案

因为它至少保证：

- 程序端数据稳定
- 图片文件稳定
- 路径稳定
- WPS 用户只需要执行固定宏

比起直接硬写 `DISPIMG` 底层结构，这条路风险更低。

## 主表约定

当前主表 `商品列表` 中，图片相关列建议约定为：

- `图片`
- `图片路径`

含义：

- `图片`：给 WPS 宏插图或后续显示
- `图片路径`：程序生成的相对路径，例如 `img/product_xxx_1_cover.jpg`

## 宏模板的定位

当前提供的宏模板定位是：

- 帮你批量读取 `图片路径`
- 把图片放到 `图片` 列
- 控制位置、大小、随单元格移动

它不是最终的 `DISPIMG` 生成器。

更准确地说，它是：

- `WPS-DISPIMG` 流程里的中间自动化工具

## 什么时候再继续追 `DISPIMG`

只有满足下面任一条件时，才继续往“程序自动生成 DISPIMG”推进：

1. 找到 WPS 官方可编程接口
2. 找到稳定可验证的宏方案
3. 找到可复用的社区方案，并能在当前环境重复验证

否则：

- 继续保持“程序导出 + WPS 宏处理”的组合方案

## 一句话

当前最稳的 `WPS` 方案不是“程序直接产出 DISPIMG”，而是：

- 程序准备好图片路径和图片文件
- WPS 宏完成图片列的批量处理
