# 项目逻辑详解

## 文档目的

这份文档不是产品介绍，也不是页面说明。

它的目标只有一个：

- 让一个没接触过这个项目的人，读完后能理解当前项目的核心逻辑
- 能判断每个页面和 Excel 的关系
- 能按同样的业务逻辑重新写一个新项目

这份文档重点讲：

- 数据从哪里来
- 页面之间怎么联动
- 各字段的默认值和覆盖规则
- 规则表如何参与计算
- 商品如何从创建页进入列表页
- Excel 如何参与读取、同步和缓存

## 一、项目整体结构

当前项目可以拆成两条主线：

### 1. 规则配置主线

由一份主 Excel 驱动。

包含：

- 预设
- 选项
- 模板规则
- 创建页的默认值和查表依据

### 2. 商品结果主线

由另一份列表 Excel 驱动。

包含：

- 创建页生成的商品记录
- 图片
- 变体
- 扩展字段
- 列表页展示和保存

一句话概括：

- 主 Excel 负责“算”
- 列表 Excel 负责“存”

## 二、页面和职责

当前核心页面有 5 个：

1. 预设页
2. 选项页
3. 模板页
4. 创建页
5. 列表页

### 1. 预设页

职责：

- 管理国家 / 平台组合
- 给每个组合维护参数项
- 提供创建页的默认参数环境

### 2. 选项页

职责：

- 维护通用下拉分组
- 给创建页和模板页提供可选项来源

### 3. 模板页

职责：

- 维护规则表
- 作为创建页查表计算的规则来源

### 4. 创建页

职责：

- 读取当前预设组合
- 读取选项分组
- 读取模板规则
- 录入商品
- 计算结果
- 生成列表记录

### 5. 列表页

职责：

- 显示已生成的商品记录
- 处理列表 Excel 的绑定、刷新、保存
- 展示图片和结果字段

## 三、核心数据模型

## 3.1 预设模型

一条预设记录本质上是：

- 一个组合
- 多个参数项

逻辑结构可以理解成：

```ts
type PresetRecord = {
  id: string
  country: string
  platform: string
  country_platform: string
  sort: number
  items: PresetItem[]
}

type PresetItem = {
  id: string
  preset_id: string
  name: string
  type: "number" | "boolean" | "rule"
  unit: string
  value: string
  rule_mode: "table" | "condition" | "advanced"
  rule_table_id: string
  rule_config: object
  sort: number
}
```

关键理解：

- `name` 是业务字段名，例如 `折扣`、`交易费率`、`是否包邮`
- `type` 决定编辑方式
- `rule` 类型不是直接存最终结果，而是存规则引用或规则配置

### 当前常用预设字段

项目里已经围绕这些字段形成固定逻辑：

- 折扣
- 活动费率
- 交易费率
- 提现费率
- 汇损
- 税率
- 贴单费用
- 是否包邮
- 卖家支付运费

其中：

- `是否包邮` 是布尔型预设
- `卖家支付运费` 是规则型预设
- 其它大多是数值型预设

## 3.2 选项模型

选项页的数据结构本质是：

```ts
type OptionGroup = {
  key: string
  title: string
  description: string
  sort: number
  enabled: number
  remark: string
}

type OptionEntry = {
  id: string
  groupKey: string
  label: string
  value: string
  sort: number
  enabled: number
  remark: string
}
```

当前默认重要分组：

- `category`
  - 中文名：`类目`
- `ad_type`
  - 中文名：`广告类型`
- `shipping_included`
  - 中文名：`是否包邮`

关键理解：

- 选项页提供的是“候选值来源”
- 创建页和模板页通过字段名去匹配选项分组

## 3.3 模板规则模型

当前模板页已经从旧的“类型专用结构”收敛到更通用的规则表结构。

一张规则表可以理解成：

```ts
type TemplateTable = {
  id: string
  name: string
  dimensions: TemplateDimension[]
  resultColumns: TemplateResultColumn[]
  records: TemplateRecord[]
  ...
}

type TemplateDimension = {
  id: string
  fieldName: string
  kind: "enum" | "range"
  ...
}

type TemplateResultColumn = {
  key: string
  label: string
  type: "text" | "number"
}

type TemplateRecord = {
  id: string
  conditions: Record<string, unknown>
  results: Record<string, unknown>
}
```

关键理解：

- 规则表不是只支持固定值、一维、二维
- 本质上是“若干维度 + 若干结果列 + 若干记录”
- `enum` 维度表示枚举匹配
- `range` 维度表示区间匹配

这意味着：

- 双枚举只是 2 个 `enum`
- 二维区间只是 2 个 `range`
- 枚举 + 区间就是混合维度
- 多枚举、多区间都可以表达

## 3.4 创建页表单模型

创建页表单主要分几块：

### 1. 预设上下文

- 国家
- 平台
- 当前选中预设组合
- 当前预设参数快照

### 2. 商品输入

- 名称
- 款号
- 成本
- 重量
- 类目
- 广告类型
- 是否包邮
- 备注

### 3. 计算输入

- 折前价格
- 折后售价
- 收入
- 利润率
- 净利润

### 4. 结果字段

- 卖家运费
- 固定附加费
- 活动费
- 交易费
- 提现费
- 汇损金额
- 税费
- 总费用

### 5. 图片和变体

- 图片数组
- 变体组数组
- 扩展字段数组

## 3.5 列表记录模型

创建页点“添加到列表”后，会生成一个完整的商品 bundle。

其中包括：

- `record`
- `images`
- `variants`
- `fields`

其中 `record` 至少包含：

- 基础商品信息
- 当前基准
- 各结果字段
- 预设快照
- 计算快照

关键理解：

- 列表页不是重新算一次
- 而是保存创建页当时的快照结果

也就是说：

- 进入列表的商品记录是“计算结果快照”
- 不是永远跟着后续规则实时重算

## 四、最关键的联动逻辑

这里是整个项目最重要的部分。

## 4.1 预设页 -> 创建页

创建页会读取当前选中的预设组合。

作用分两类：

### 1. 提供默认值

例如：

- 折扣
- 活动费率
- 交易费率
- 提现费率
- 税率
- 贴单费用
- 是否包邮

### 2. 提供规则入口

例如：

- 卖家支付运费

这个字段不是最终数值，而是一个规则型参数，通常引用某张模板表。

### 当前行为

创建页打开或切换预设后：

- 会读取当前预设组合的参数项
- 把“预设驱动字段”作为默认值
- 用户仍然可以在创建页继续改

最重要的规则是：

- 预设只负责给默认值
- 创建页允许覆盖

## 4.2 选项页 -> 创建页

创建页中如果某个字段被定义成选项字段，就不再手写选项。

当前典型字段：

- 类目
- 广告类型
- 是否包邮

创建页会做的事：

- 先根据字段 key 找字段注册表
- 再从字段注册表找到对应 option group
- 最后从选项 store 拿下拉项

关键理解：

- 创建页的下拉不是页面本地写死
- 是从选项页驱动过来的

## 4.3 选项页 -> 模板页

模板页里如果某个维度字段名和选项页分组同名，就自动把它理解成可选枚举。

例如：

- 维度字段名写 `类目`
- 系统发现选项分组里有 `类目`
- 那么这列就适合做下拉

如果字段名没有对应选项分组：

- 例如 `折后售价`
- 例如 `重量`

那就按普通输入或区间值来处理。

关键理解：

- 模板页不是手动绑定技术 key
- 现在是尽量按中文字段名走

## 4.4 模板页 -> 预设页

预设里的 `rule` 类型参数项可以引用模板规则表。

最典型的是：

- 卖家支付运费

预设页里不存完整规则，而是存：

- 当前规则模式
- 引用的规则表 ID
- 规则配置

也就是说：

- 模板页维护“规则内容”
- 预设页维护“在哪个组合里启用哪张规则表”

## 4.5 模板页 -> 创建页

创建页计算时会读取模板表，并按当前输入去查值。

当前已经接到通用查表运行时的，是：

- 卖家支付运费

核心流程：

1. 从预设项里找到 `卖家支付运费`
2. 取出它绑定的规则表 ID
3. 拿到模板页当前规则表
4. 按维度顺序组装参数
5. 命中一条记录
6. 取结果列值

当前卖家运费典型维度是：

- 折后售价
- 重量

但底层已经改成通用查表模型，不再强依赖老的二维区间结构。

## 4.6 创建页内部字段联动

这是最容易重写错的部分。

### 1. 当前基准概念

创建页支持多个输入字段作为“当前反推基准”：

- 折前价格
- 折后售价
- 收入
- 利润率
- 净利润

也就是说：

- 用户当前在改哪个字段
- 系统就按那个字段作为反推入口

### 2. 计算方向

如果当前基准是：

- `折前价格`
  - 则先根据折扣算出折后价
- `折后售价`
  - 则直接以折后价为基准
- `收入`
  - 当前实现里收入视为折后价
- `利润率`
  - 则通过公式反推折后价
- `净利润`
  - 则通过公式反推折后价

### 3. 结果快照

创建页最终会生成一个 `calculationSnapshot`。

里面会有：

- 折扣、费率类参数
- 折前价、折后价
- 各费用项
- 总费用
- 收入
- 净利润
- 利润率
- 当前是否包邮
- 预设是否包邮

这个快照既用于：

- 右侧结果展示
- 运算过程弹窗
- 添加到列表时写入记录

### 4. 包邮逻辑

当前有两个层面的“是否包邮”：

- 预设里的默认值
- 创建页当前商品的实际值

项目当前采用的逻辑是：

- 预设页里的 `是否包邮` 提供默认值
- 创建页里仍然展示 `是否包邮` 字段
- 创建页的值可以修改
- 真正参与计算的，是创建页当前值

在结果区会同时保留：

- 当前是否包邮
- 预设是否包邮

这样做的目的：

- 让用户知道当前商品是否沿用了预设
- 或者已经偏离预设

### 5. 卖家运费逻辑

当前卖家运费的逻辑是：

1. 先看当前商品 `是否包邮`
2. 如果是 `是`
   - 卖家运费直接为 `0`
3. 如果不是
   - 再去查预设绑定的规则表
4. 如果规则表不存在或没命中
   - 当前按 `0` 处理

也就是说优先级是：

1. 当前商品是否包邮
2. 预设绑定的卖家运费规则
3. 无规则时回落 0

### 6. 固定附加费逻辑

当前固定附加费仍然保留为结果字段之一。

它可以来自：

- 用户输入
- 模板规则

但本质上都应该被纳入总费用。

### 7. 总费用和利润逻辑

当前核心计算链路可以理解成：

1. 先得到折后价
2. 根据费率算活动费、交易费、提现费、汇损金额、税费
3. 加上贴单费、卖家运费、固定附加费
4. 得到总费用
5. 收入当前等于折后价
6. 净利润 = 收入 - 总费用 - 成本
7. 利润率 = 净利润 ÷ 收入

当前页面里还有运算过程展示，会把这些步骤拆出来说明。

## 五、列表页相关逻辑

## 5.1 创建页 -> 列表页

创建页点击“添加到列表”后：

- 不是立即写 Excel
- 而是先生成程序内部的商品 bundle
- 然后进入列表 store

关键理解：

- “加入列表”是生成正式记录
- “保存列表 Excel”是落盘

## 5.2 列表记录为什么要存快照

列表记录不是只存商品基本信息。

它还会保留：

- 计算快照 JSON
- 预设快照 JSON
- 当前基准
- 费用结果

这样做的目的是：

- 列表页可以直接展示当时的结果
- 不依赖后续预设或模板变化实时重算

这对业务是重要的，因为：

- 一个商品进入列表时看到的结果，就是当时确认的结果

## 5.3 列表页图片逻辑

列表页只显示封面图。

封面图来源：

- 商品图片数组里 `isCover = 1` 的那张
- 如果没有明确 cover，就取第一张

列表页支持：

- 图片缩略图
- 点击预览
- 缩放
- 拖拽平移

## 5.4 列表页与 Excel

列表页绑定的是第二份 Excel。

它负责：

- 读回商品记录
- 读回图片关系
- 读回变体关系
- 读回扩展字段
- 保存当前列表到 Excel

关键理解：

- 列表 Excel 是商品数据文件
- 不是规则文件

## 六、Excel 结构和同步逻辑

## 6.1 规则 Excel

主 Excel 当前承载：

- 预设相关 sheet
- 选项相关 sheet
- 模板相关 sheet

更准确地说：

- 预设页、选项页、模板页都共用同一份主 Excel

### 预设相关

预设现在是两层结构：

- `preset_groups`
- `preset_items`

### 选项相关

选项现在是两层结构：

- option groups
- option entries

### 模板相关

模板当前至少涉及：

- 规则表主信息
- UI 配置
- 维度
- 结果列
- 记录内容

## 6.2 列表 Excel

列表 Excel 当前承载：

- 商品主表
- 商品记录表
- 图片关系
- 变体关系
- 扩展字段
- 程序信息

关键理解：

- 主表是给用户看的
- 记录表和关系表是给程序恢复完整数据的

## 6.3 同步方式

### 主 Excel

预设、选项、模板当前都是：

- 从主 Excel 读取
- 修改内存数据
- 再整本写回

### 列表 Excel

列表页也是：

- 从列表 Excel 读取
- 修改列表 store
- 保存时整本写回

## 6.4 自动同步和手动同步

### 预设、模板、选项

在 Tauri 且绑定真实 Excel 路径时：

- 修改后会走防抖自动同步

### 列表

列表页主要还是手动保存。

原因：

- 列表记录是正式数据
- 不适合边改边频繁覆盖文件

## 七、缓存逻辑

## 7.1 预设缓存

预设 store 会缓存：

- 预设记录
- 当前选中组合
- 已绑定 Excel 文件名
- 已绑定 Excel 路径

作用：

- 刷新不丢组合数据
- Tauri 下可恢复路径绑定

## 7.2 创建页缓存

创建页会缓存草稿。

作用：

- 跳去别的页面再回来不丢
- 录到一半不丢

当前重要行为：

- 用户点重置时，会清空当前输入
- 但预设驱动字段会回到当前预设默认值

## 7.3 列表缓存

列表 store 会缓存：

- 记录
- 图片
- 变体
- 扩展字段
- 绑定文件名
- 绑定路径
- dirty 状态
- 上次同步时间

作用：

- 页面切换不丢列表
- 重开程序后可以继续工作

## 八、目前已经统一掉的关键配置层

如果要重写，新项目非常建议保留这些“统一注册表”思想。

### 1. 字段注册表

当前已经有统一字段注册表去描述这些字段：

- 类目
- 广告类型
- 是否包邮

注册表里统一了：

- formKey
- 中文字段名
- optionSource
- optionGroupKey

作用：

- 创建页
- 选项页
- 模板页

对同一个字段有统一理解。

### 2. 创建页配置

当前创建页已经把一部分容易散落的东西收到了常量里，例如：

- 预设字段定义
- 结果区字段配置
- 费用区字段配置
- 当前基准字段配置

新项目建议继续这么做，不要再把字段名散在页面里。

### 3. 列表字段配置

当前已经新增统一列表字段配置层。

它统一了：

- 列表页显示列
- Excel 主表列
- Excel 记录表 key
- 字段别名
- 宽度
- 类型

这是后续可维护性的关键点。

## 九、已知风险和重写时需要注意的坑

## 9.1 最重要的风险：Excel 整本重写

当前 Excel 写回方式仍然是整本重写。

这意味着：

- 对带图片、drawing、媒体资源的工作簿有风险

如果重写项目，最好尽早定清：

- Excel 只是配置源
- 还是要做高保真回写

如果要求高保真保留图片和媒体关系，不能长期依赖简单整本重写。

## 9.2 “当前值”和“默认值”要严格区分

例如 `是否包邮`：

- 预设页里的是默认值
- 创建页里的是当前商品实际值

如果重写时把它们混成一个字段，会让很多联动出错。

正确做法是始终保留两层概念：

- default
- current

## 9.3 规则表不要再写死类型分支

当前虽然界面上还会出现某些规则类型名称，但底层已经往“维度驱动”走。

重写时建议直接采用：

- dimensions
- resultColumns
- records

不要再把：

- 一维
- 二维
- 双枚举
- 枚举 + 区间

做成一堆互相独立的数据结构。

## 9.4 列表记录不要实时重算

当前列表记录是快照模式。

这点很重要。

如果新项目改成列表页实时重算，会带来这些问题：

- 旧商品结果随规则变化而变化
- 无法复现当时的计算结果
- 商品记录不稳定

除非业务明确要求，否则不建议改。

## 十、建议的重写顺序

如果要重写项目，不建议先抄页面，而建议按下面顺序：

1. 先定数据模型
   - 预设
   - 选项
   - 模板
   - 创建草稿
   - 列表记录
2. 再定 Excel 结构
   - 主 Excel
   - 列表 Excel
3. 再写 store / service 层
   - 读取
   - 标准化
   - 查表
   - 保存
4. 再写创建页计算引擎
5. 最后写页面

理由很简单：

- 这个项目真正复杂的不是 UI
- 而是字段联动、规则查表、Excel 同步、快照边界

## 十一、给接手人的一句话总结

这个项目真正的核心不是“5 个页面”，而是下面这条链路：

- 主 Excel 提供预设、选项、模板规则
- 创建页把这些规则和当前商品输入组合起来完成测算
- 列表页把测算结果以快照形式沉淀到第二份 Excel

只要把这条链路理解清楚，就能重写；如果只抄页面外观，不理解这条链路，项目一定会写歪。
