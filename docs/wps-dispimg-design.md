# WPS `DISPIMG` 方案

## 结论

如果这个项目的主要用户是 `WPS` 用户，而且列表页导出的 Excel 主要在 `WPS` 里查看和使用，那么图片方案不应该再优先押在：

- `rust_xlsxwriter embed_image()`
- `exceljs`
- 其它通用 xlsx 图片插入库

更合适的方向是：

- `WPS + DISPIMG`

也就是：

- 程序负责输出商品表和图片资源
- `WPS` 负责把图片以它自己的“嵌入单元格图片”机制显示出来

## 为什么改成 `DISPIMG`

原因有 3 个：

### 1. `DISPIMG` 是 WPS 原生机制

WPS 社区和问答里明确提到：

- `DISPIMG` 用于在单元格中显示图片
- 它是 WPS 侧图片嵌入/显示体系的一部分

参考：

- https://bbs.wps.cn/topic/73968
- https://bbs.wps.cn/topic/9628

### 2. `rust_xlsxwriter` 的 `embed_image()` 对应的是 Excel 365 新语义

`rust_xlsxwriter` 官方文档写得很明确：

- “Embedding an image in a cell”
- 对应的是 Excel 365 2023+ 的 “Place in Cell”

参考：

- https://rustxlsxwriter.github.io/examples/embedded_images.html
- https://docs.rs/rust_xlsxwriter/latest/rust_xlsxwriter/cookbook/index.html

这意味着：

- 它更适合 `Microsoft Excel`
- 不等于 `WPS` 一定按同样方式表现

### 3. WPS 社区里已经能看到“批量导入/导出嵌入单元格图片”的独立讨论

这说明：

- WPS 的嵌入单元格图片确实是单独体系
- 不能简单等同于“普通插图”
- 在工具层面也和 Excel 通用库存在差异

参考：

- https://bbs.wps.cn/topic/52472
- https://bbs.wps.cn/topic/51529
- https://bbs.wps.cn/topic/42604

## 最关键的判断

不要试图在程序里“纯手搓” `DISPIMG("ID_xxx", 1)` 的底层资源结构。

原因：

- 社区里很多人在问 `ID_xxx` 是怎么生成的
- 公开资料里没有看到稳定、完整、官方的文件结构说明
- 自己逆向去伪造这套结构，维护风险很高

参考：

- https://bbs.wps.cn/topic/9628

所以更稳的思路是：

- 不去伪造 `DISPIMG` 的内部结构
- 而是借助 `WPS` 自己的能力来生成/转换

## 推荐架构

建议做成两层：

### 第一层：程序导出标准商品表

程序继续负责：

- `商品列表`
- `图片关系`
- `变体关系`
- `扩展字段`
- 图片文件目录

也就是当前列表 Excel 体系。

### 第二层：WPS 侧把图片转成 `DISPIMG`

由 `WPS` 来完成：

- 根据图片路径插入图片
- 生成 WPS 自己的嵌入图片资源
- 在图片列写入 `DISPIMG(...)`

这样就把最难、最不稳定的部分交给 WPS 自己。

## 项目里应该怎么改

## 主表设计

主表仍然保留：

- `名称`
- `款号`
- `图片`
- `变体`
- `国家`
- `平台`
- `类目`
- `广告类型`
- `是否包邮`
- `成本`
- `重量(g)`
- `境内运费`
- `折后价格(R$)`
- `折前价格(R$)`
- `利润率`
- `净利润(R$)`
- `收入(R$)`
- `活动费`
- `交易费`
- `佣金费`
- `提现费`
- `汇损`
- `税费`
- `贴单费`
- `固定附加费`
- `备注`

另外建议新增一个内部用列：

- `图片路径`

这个列可以：

- 放在主表靠后位置
- 或者放在单独 sheet

作用是给 WPS 自动化脚本用。

## 图片资源如何存

推荐：

- 列表 Excel 同级目录建立 `img/`
- 程序保存时把商品图片复制到这个目录
- 主表 `图片路径` 列写相对路径

例如：

```txt
商品列表.xlsx
img/
  product_001_cover.jpg
  product_001_red.jpg
```

主表里：

```txt
图片路径 = img/product_001_cover.jpg
```

这样好处很明显：

- Excel 可以单独移动
- 图片资源跟 Excel 一起走
- WPS 宏也更容易按路径读取

## 程序当前最适合负责的部分

程序现在最适合负责：

1. 生成 `商品列表.xlsx`
2. 写好所有业务字段
3. 写好 `图片路径`
4. 写好 `图片关系`
5. 把封面图复制到 Excel 同级 `img/`

也就是说：

- 程序先把“数据”和“图片文件”准备好
- `DISPIMG` 的最后一步交给 WPS

## `DISPIMG` 方案的落地方式

### 方案 A：用户在 WPS 中执行宏

程序导出后：

- 用户点一个 `WPS 批量嵌图` 宏
- 宏读取 `图片路径` 列
- 为每行生成 WPS 嵌入单元格图片
- 最终主表图片列显示为 `DISPIMG`

优点：

- 最稳
- 最贴近 WPS 原生行为

缺点：

- 需要用户环境支持 WPS 宏

### 方案 B：程序调用 WPS 自动化

如果目标环境允许：

- Windows
- 本机安装 WPS
- 可调用 COM / 宏 / 脚本

那可以由程序直接触发：

- 打开 WPS
- 执行指定宏
- 完成批量嵌图

优点：

- 用户操作更少

缺点：

- 平台限制更强
- 自动化链路更复杂

## 不建议做的事

### 1. 不建议继续把“WPS 嵌图”主逻辑放在通用 xlsx 库里

包括：

- `rust_xlsxwriter`
- `exceljs`
- `SheetJS`

这些库可以保留做：

- 普通 sheet 导出
- 文本和数字写入

但不要再把它们当作 WPS `DISPIMG` 的主力方案。

### 2. 不建议自己逆向构造 `ID_xxx`

原因：

- 资料不透明
- 风险高
- 后期维护成本太大

## 推荐的实施顺序

### 第一阶段

- 列表 Excel 主表增加 `图片路径`
- 保存时把图片复制到 Excel 同级 `img/`
- Rust 继续负责列表 Excel 数据导出
- 暂时不再强求程序直接写出 `DISPIMG`

### 第二阶段

- 增加一份 WPS 宏模板
- 宏根据 `图片路径` 列批量转换图片列

### 第三阶段

- 如果环境允许，再考虑程序直接调用 WPS 自动化

## 最后的建议

如果你的目标是：

- `WPS` 用户优先
- 视觉和使用效果优先
- 图片在 WPS 里必须像原生嵌入单元格一样工作

那最稳的路线就是：

- 程序准备数据
- WPS 负责生成 `DISPIMG`

## 一句话结论

对于这个项目，`DISPIMG` 更像是 “WPS 原生显示层”，而不是一个适合被我们在 Rust/Node 里直接伪造的底层格式。

程序应该负责：

- 商品数据
- 图片文件
- 图片路径
- 关系表

然后把最后一步“转成 WPS 的嵌入单元格图片”交给 WPS 自己完成。
